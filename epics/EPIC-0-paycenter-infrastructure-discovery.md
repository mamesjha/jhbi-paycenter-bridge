## Summary

Before any code is written, we must understand Jack Henry PayCenter's raw event infrastructure. We are **not** using the Enterprise Event System (EES) abstraction layer. Instead, we want direct access to the raw payment events generated by PayCenter for Zelle, FedNow, and RTP.

This epic covers all discovery, documentation, and validation work required to establish what events are available, how to subscribe to them, and what the payloads look like per rail type.

## Goal

Produce a validated, documented understanding of PayCenter's raw event infrastructure sufficient to begin implementation of the bridge service. All subsequent epics are blocked until this epic is complete.

## Why Not EES?

Jack Henry's Enterprise Event System (EES) publishes a normalized, abstracted Event 800 wrapper across all rails. We require the **raw, pre-abstraction events** per rail so that:

- We receive the full native payload from each network (Zelle/EWS, FedNow/ISO 20022, RTP/ISO 20022)
- We control schema evolution and field mapping ourselves
- We are not dependent on JH's normalization choices or EES availability
